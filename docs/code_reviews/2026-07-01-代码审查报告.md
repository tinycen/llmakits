## 🔴 严重bug (CRITICAL)

### 1. `eval()` 和裸 `except:` 双重危险 — 异常被静默吞没

**文件**: `llmakits/message/formatter.py` 第 97–104 行

```python
try:
    if processed_text.startswith("```json"):
        processed_text = processed_text.strip("```json\n")

    converted_json = json.loads(processed_text)
except json.JSONDecodeError:
    try:
        converted_json = eval(processed_text)  # 危险！
    except:                                    # 危险！
        try:
            extracted_json = extract_json_from_string(processed_text)
            if extracted_json:
                converted_json = json.loads(extracted_json)
        except:                                # 危险！
            pass
```

**问题说明**:

- `eval()` 对任意字符串求值，如果 LLM 返回的文本中包含恶意代码（如 `__import__('os').system('...')`），`eval()` 会直接执行，造成命令注入风险。
- 三层嵌套的裸 `except:` 块会捕获所有异常（`KeyboardInterrupt`、`SystemExit` 等），且最终 `pass` 导致解析失败时完全静默，调用方收到 `None` 后只能靠外层 `if converted_json is not None` 判断，容易引发后续 `None` 的 `['answer']` 访问。

**修复建议**:

```python
import ast

try:
    if processed_text.startswith("```json"):
        processed_text = processed_text.strip("```json\n")
    converted_json = json.loads(processed_text)
except json.JSONDecodeError:
    try:
        # 用 ast.literal_eval 替代 eval，仅支持 Python 字面量，安全无害
        converted_json = ast.literal_eval(processed_text)
    except (ValueError, SyntaxError):
        try:
            extracted_json = extract_json_from_string(processed_text)
            if extracted_json:
                converted_json = json.loads(extracted_json)
        except (ValueError, json.JSONDecodeError):
            converted_json = None

if converted_json is not None:
    # ... 处理逻辑
else:
    raise Exception("format_json_error, 无法解析为json格式")
```

**文件**: `llmakits/dispatcher.py` 第 325、358、361 行

```python

### 3. `extra_body` 嵌套结构逻辑容易出错

**文件**: `llmakits/load_model.py` 第 91–108 行

```python
if extra_body_nested:
    if extra_body:
        params['extra_body'] = {**extra_body, "extra_body": extra_body_nested}
    else:
        params['extra_body'] = {"extra_body": extra_body_nested}
elif extra_body:
    params['extra_body'] = extra_body
```

**问题说明**: 当同时存在 `extra_body` 和 `extra_body_nested`（即 `extra_enable_thinking`）时，最终结构为 `{"extra_body": {...}, "extra_body": {...}}` —— 两个同名字段，后者覆盖前者。这看起来是一个双层嵌套设计（`extra_body.extra_body`），但最终传给 API 的参数行为是否符合预期需要充分测试。

**修复建议**: 明确设计意图，如果只需要一层嵌套，考虑展平；如果需要双层嵌套，使用不同的 key（如 `nested_options`）避免歧义，并在代码注释中说明其必要性。

## 审查备注

### Pyright 类型检查

Pyright 报告 15 个诊断，其中大部分为第三方依赖（`funcguard`、`filekits`、`zai` 等）无法在当前环境中解析导致的导入错误，非代码本身问题。真正的类型相关问题为：

- `tests/test_image_retry_logging.py` 第 293 行：`result.error.get_error_message()` — Pyright 认为 `result.error` 可能为 `None`（`Optional[ResponseError]`），建议先做空值判断。

### 配置一致性问题

- `setup.py`: `python_requires='>=3.10'`
- `pyrightconfig.json`: `"pythonVersion": "3.11"`

两者不一致，建议统一为 `3.10` 或更高。

### 正面评价

1. **完善的测试覆盖**: `tests/test_image_retry_logging.py` 使用 `unittest` + `unittest.mock` 对图片重试、缓存等关键路径进行了单元测试，覆盖面较好。
2. **清晰的模块划分**: `dispatcher`、`llm_client`、`message`、`utils` 等模块职责分明。
3. **LRU 缓存设计**: `ImageBase64Cache` 实现合理，支持单图缓存和图片组缓存，并记录失败 URL 避免重复下载。
4. **全局重试状态共享**: `_GLOBAL_RETRY_STATE` 作为模块级全局状态，通过快照方法暴露给外部，避免直接修改内部状态。
5. **日志记录规范**: 使用 `funcguard` 的 `setup_logger` 统一日志输出，包含模型名称和错误标签，便于排查。
