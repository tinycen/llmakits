### 🔴 严重 Bug（可能导致运行时错误或逻辑错误）

#### 1. `dispatcher.py` — `validate_func` 对 falsy 合法返回值误判为失败
**位置**：`dispatcher.py` 的 `execute_task` 方法，验证逻辑分支
**问题代码**：
```python
if is_valid:
    if validated_value:  # ❌ Bug：空字符串、0、False、None、[] 等合法值会被当作失败
        if return_detailed:
            return ExecutionResult(...)
        return validated_value, total_tokens
    else:
        content = "输出结果：条件校验失败, trying next model ..."
        # 会错误地切换到下一个模型
```
**影响**：当验证函数返回 `(True, "")` 或 `(True, 0)` 等 falsy 但合法的结果时，调度器会误判为校验失败并切换到下一个模型。
**修复建议**：改为 `if validated_value is not None:` 或移除该判断，直接信任 `is_valid` 的返回值。

#### 2. `retry_handler.py` — `message_config["include_img"]` 可能引发 `KeyError`
**位置**：`retry_handler.py` 的 `handle_rate_limit_error` 和 `should_retry_for_image_error`
**问题代码**：
```python
if any(...) and message_config["include_img"]:  # ❌ 若 message_config 不含该键直接崩溃
```
以及
```python
should_retry_for_image_error(...) and message_config["include_img"]
```
**影响**：如果调用者传入的 `message_info` 字典不包含 `"include_img"` 键，触发限流或图片错误时会直接抛出 `KeyError`，中断整个调度流程。
**修复建议**：全部改为 `message_config.get("include_img", False)`。

---

#### 3. `image_cache.py` — `mark_failed()` 无法记录新的失败 URL
**位置**：`image_cache.py` 的 `mark_failed` 方法
**问题代码**：
```python
def mark_failed(self, url: str, reason: str = "") -> None:
    if not url:
        return
    if url in self.failed_cache:
        self.failed_cache.move_to_end(url)
        self.failed_cache[url] = reason
        if len(self.failed_cache) > self.max_size:
            self.failed_cache.popitem(last=False)
    # ❌ 缺少 else 分支：URL 不在缓存中时，永远不会被记录！
```
**影响**：首次下载失败的图片 URL 不会被记入失败缓存，导致后续重复尝试下载，失去缓存避免重复失败的意义。
**修复建议**：添加 `else` 分支将新 URL 写入 `failed_cache`。

#### 4. `dispatcher.py` — `_remove_model()` 只删除第一个匹配组中的模型
**位置**：`dispatcher.py` 的 `_remove_model` 方法
**问题代码**：
```python
for group_name, group_models in self.model_groups.items():
    new_group_models = []
    for model in group_models:
        if model['sdk_name'] != sdk_name or model['model_name'] != model_name:
            new_group_models.append(model)
    self.model_groups[group_name] = new_group_models
    return  # ❌ 找到第一个匹配组后就 return，其他组中的同名模型不会被删除
```
**影响**：如果同一个 `(sdk_name, model_name)` 出现在多个模型组中，API 密钥耗尽后只会从一个组中移除，其他组仍会尝试调用已耗尽的模型。
**修复建议**：移除 `return`，让循环遍历完所有组。

#### 5. `retry_handler.py` — `preprocess_message_info()` 存在空指针风险
**位置**：`retry_handler.py` 的 `preprocess_message_info` 方法
**问题代码**：
```python
self.image_cache = None
try:
    from ..dispatcher import ModelDispatcher
    self.image_cache = ModelDispatcher.get_image_cache()
except ImportError:
    pass

# 后续直接调用：
img_list = resolve_images_with_cache(..., self.image_cache, ...)  # ❌ 可能为 None
```
**影响**：若因循环引用导致 `ImportError`，`self.image_cache` 为 `None`，后续传入 `resolve_images_with_cache` 或 `convert_images_to_base64` 时会抛出 `TypeError/AttributeError`。
**修复建议**：在使用前检查 `if self.image_cache is None: return message_info`（或做其他兜底处理）。

### 🟡 中等问题（逻辑缺陷或不良实践）

#### 6. `dispatcher.py` — `printed_model_indices` 逻辑混乱导致日志缺失
**位置**：`dispatcher.py` 的 `execute_task` 与 `_print_next_model_info`
**问题**：`_print_next_model_info` 会把「下一个模型」的索引加入 `printed_model_indices`。而 `execute_task` 在打印当前模型错误时，若当前索引已在集合中，则跳过打印当前模型头信息。
**影响**：模型 A 失败后打印了「下一个模型 B」的信息；当模型 B 真的失败时，由于 `idx=1` 已在集合中，**模型 B 的错误日志头信息被吞掉**，用户看不到是哪个模型失败了。

---

#### 7. `llm_client.py` — 裸 `except` 捕获所有异常
**位置**：`llm_client.py` 的 `_handle_response`
**问题代码**：
```python
try:
    result = response.choices[0].message.content
except:  # ❌ 捕获所有异常，包括 KeyboardInterrupt、SystemExit 等
    result = response.choices[0].delta["content"]
```
**影响**：极端情况下可能掩盖本应向上抛出的系统级异常。
**修复建议**：

精确判断：

- 通过 IndexError 判定 error_tag = "原始响应中没有choices"；
- 新增一个 response_error 用于 AttributeError。
- 新增 else 用于输出兜底 response_error。

---

#### 8. `llm_client.py` — `process_stream_response` 缺少防御式编程
**位置**：`llm_client.py` 顶部
**问题代码**：
```python
def process_stream_response(response):
    result = ""
    for chunk in response:
        result += chunk.choices[0].delta.content  # ❌ 若 choices 为空或 delta 为 None 直接崩溃
    return result
```
**影响**：某些 LLM 提供商的流式 chunk 可能不含 `choices` 或 `delta`，会导致 `IndexError/AttributeError`。
可参考 7. `llm_client.py` — 裸 `except` 捕获所有异常 修复建议。
建议：是否可尝试 提取统一 response 检测方法？方便多处调用。

#### 9. `llm_client.py` — 重试计数与 API Key 切换的交互缺陷
**位置**：`llm_client.py` 的 `send_message`
**问题**：`api_retry_count` 在每次 `should_retry=True` 时递增，但**切换 API Key 并不会重置计数**。
**影响**：假设有 3 个 API Key，每个 Key 因限流失败 1 次，第 4 次重试时（哪怕还有新 Key），`api_retry_count` 已达 4，直接跳出循环抛出 `API_RETRY_REACHED`，**剩余 Key 未被充分利用**。

**临时修复建议**：
在load模型的阶段，就对 API Key 的数量进行检测，当 > 3 时，直接打印相关提醒（> 3 个 API Key 无法充分利用 ）。
